## 一、项目目标与核心流程

**目标**：为高校 Python 程序设计课程提供一个可演示的个性化学习多智能体系统，覆盖「对话式画像 → 知识路径规划 → 个性化资源生成 → 多 Agent 辩论审核 → 代码实操 → 苏格拉底辅导 → 学习效果评估」完整闭环。

**核心流程（Mermaid 语义）**：

```text
学生输入
   │
   ▼
Orchestrator（意图识别 + 会话状态维护）
   │
   ├── 聊天/问答 ──► Profiler ──► Navigator ──► Generator ──► Reviewer
   │                                              │
   │                                              ▼
   │                                         资源包 + 辩论报告
   │                                              │
   ▼                                              ▼
前端展示 ◄────────────────────────────── 代码实操 / 辅导 / 评估
```

## 七、代码规范速查（用于团队协作）

### 7.1 Python 后端

1. **模块注释**：每个 `.py` 文件顶部必须包含模块 docstring，说明：
   - 需求/功能；
   - 主要类/函数职责；
   - TODO 清单（`[已完成]` / `[待完成]`）。
2. **命名**：类名 `PascalCase`，函数/变量 `snake_case`，常量 `UPPER_SNAKE_CASE`。
3. **类型注解**：函数参数与返回值尽量使用 `typing` 注解；复杂结构优先用 Pydantic 模型。
4. **错误处理**：对外部调用（LLM、数据库、图存储）必须 `try/except` 并记录日志，返回降级结果。
5. **测试**：新增功能必须配套测试；修改后运行 `pytest -q`。
6. **配置**：敏感信息（API Key、密码）必须走 `.env`，禁止硬编码。

### 7.2 TypeScript / React 前端

1. **组件注释**：每个业务组件顶部添加 JSDoc，说明 props 主要用途与 TODO。
2. **命名**：组件 `PascalCase`，hooks/useStore `camelCase` 且以 `use` 开头。
3. **类型**：优先定义接口/类型，减少 `any`；后端返回类型与 `services/api.ts` 对齐。
4. **状态管理**：局部状态用 `useState/useReducer`，跨组件状态放 `zustand` store。
5. **构建**：提交前必须 `npm run build` 通过。

### 7.3 Git 工作流

- 分支：`main` 保持可演示；功能从 `main` 切 `feature/xxx`。
- 提交信息格式：
  - `feat: 新增 xxx`
  - `fix: 修复 xxx`
  - `docs: 更新 xxx`
  - `test: 补充 xxx 测试`
  - `refactor: 重构 xxx`
- 每次提交前更新相关 docstring/TODO/进度文档。
- 禁止在代码中提交真实 API Key。
