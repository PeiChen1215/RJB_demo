# 智学蜂巢 EduHive —— 三人分工与执行手册

> **生效日期**：2026-06-23  
> **适用周期**：决赛前 30 天冲刺阶段  
> **版本**：v1.3（2026-07-03 更新 Command Center 真实数据闭环与组件拆分）

---

## 一、总体原则

1. **本周任务本周验收**：每周五晚 21:00 开 30 分钟站会，逐条验收本周交付物。
2. **主链路优先**：所有开发必须先保证“登录 → 画像 → 路径 → 资源 → 辩论 → 代码 → 辅导 → 报告”这条 Demo 主链路不崩。
3. **文档同步**：每完成一个功能，必须同步更新接口文档/数据库文档/Demo 脚本中的对应部分。
4. **Git 纪律**：每天下班前 commit + push；功能分支命名：`feature/功能名`；修复分支：`fix/问题名`。
5. **阻塞立刻升级**：如果某项任务卡住超过 4 小时，立刻在群里 @ 队友求助。

## 八、Git 协作策略

### 8.1 总体原则

**`main` 分支必须随时可演示。** 所有人从 `main` 切出 `feature/*` 分支开发，本地测试通过后通过 Pull Request（PR）合并回 `main`。

**不推荐的做法：**

| 方案 | 为什么不推荐 |
|------|-------------|
| 每人一条长期个人分支 | 容易变成三条平行宇宙，合并时冲突爆炸 |
| 直接向 `main` push | 没有 review，一人推崩全队 Demo 完蛋 |

---

### 8.2 分支模型

```
main (永远可演示)
  │
  ├── feature/agent-refactor       ← 成员 A
  │
  ├── feature/db-schema            ← 成员 B
  │
  ├── feature/heatmap-ui           ← 成员 C
  │
  └── feature/socratic-chain       ← 成员 A
```

| 分支 | 用途 | 规则 |
|------|------|------|
| `main` | 稳定分支，随时能跑 Demo | 不能直接 push，必须通过 PR 合并 |
| `feature/xxx` | 单个功能开发 | 从 `main` 切出，开发完即删 |
| `hotfix/xxx` | 紧急修复 `main` 上的 Bug | 从 `main` 切出，修完立即合并 |

---

### 8.3 分支命名规范

```text
feature/agent-5-role-layering
feature/db-add-tables
feature/bkt-simplified
feature/heatmap-component
feature/socratic-chain
feature/debate-cache-degradation
feature/variable-visualization

hotfix/sse-progress-stuck
hotfix/login-password-hash
```

- 全部小写，用 `-` 连接
- 前缀必须是 `feature/` 或 `hotfix/`
- 名字要能一眼看出改了什么

### 8.4 完整工作流程

#### 1. 开始新功能前：先同步 main

```powershell
cd D:\Gitproject\RJB_Demo
git checkout main
git pull origin main
```

#### 2. 切出 feature 分支

```powershell
git checkout -b feature/agent-5-role-layering
```

#### 3. 开发 + 本地测试

```powershell
git add .
git commit -m "feat(agent): 5 角色 Agent 分层重构完成"
```

**提交前必须做：**

- 后端：跑相关测试或 `python -m pytest`
- 前端：`npm run build` 必须通过
- 如果改动影响主链路，本地把 Demo 跑一遍

#### 4. 推到远程

```powershell
git push -u origin feature/agent-5-role-layering
```

#### 5. 提 Pull Request（PR）

去 GitHub：`https://github.com/PeiChen1215/RJB_demo/pulls`

- **Base**：`main`
- **Compare**：你的 `feature/xxx`
- **标题**：`feat: 5 角色 Agent 分层重构`
- **描述**：写清楚改了什么、怎么测的、有没有影响其他模块

#### 6. Code Review

审查人检查：

- 代码会不会影响 Demo 主链路
- 有没有把 `.env`、日志、截图误提交
- 有没有明显 Bug

**审查人规则：**

- 后端改动 → 另一个后端成员审
- 前端改动 → 至少一个人审
- 数据表改动 → 全体成员知情

#### 7. 合并到 main

审查通过 + 本地测试通过 → 点击 **Merge pull request**。

合并后删除 feature 分支：

```powershell
git checkout main
git pull origin main
git branch -d feature/agent-5-role-layering
git push origin --delete feature/agent-5-role-layering
```

### 8.5 冲突解决：rebase vs merge

**推荐：feature 分支内用 `rebase`，合并到 main 时用 `merge`。**

#### feature 分支同步 main 最新代码

```powershell
git checkout feature/agent-5-role-layering
git fetch origin
git rebase origin/main
```

有冲突就解决，然后：

```powershell
git add .
git rebase --continue
```

#### 合并到 main

用 GitHub 的 **Create a merge commit**，保留 feature 分支历史。

## 九、代码与注释规范

为了让三位成员能**快速看懂彼此代码对应的需求和实现**，所有代码必须遵守以下规范。该规范已同步写入 `docs/设计骨架与实现清单.md` 第七章。

### 9.1 Python 后端规范

1. **模块级 docstring（每个 `.py` 文件顶部必须写）**
   - 第一句话说明该模块解决什么需求；
   - 列出主要类/函数职责；
   - 用 TODO 清单标明已完成/待完成项。

   示例：

   ```python
   """ReviewerAgent：资源审核、苏格拉底辅导与学习评估的统一入口。

   需求：
   - 对 Generator 产出的资源包进行多视角审核；
   - 学生答错时提供苏格拉底式提问链；
   - 根据练习/代码行为评估学习效果。

   TODO:
   - [已完成] 4-Prompt 辩论议会 + fast/full 模式
   - [已完成] SQLite 缓存与修订后重审
   - [待完成] 超时熔断与重试
   """
   ```

2. **命名规范**
   - 类名：`PascalCase`
   - 函数/变量：`snake_case`
   - 常量：`UPPER_SNAKE_CASE`

3. **类型注解**
   - 函数参数和返回值尽量使用 `typing` 注解；
   - 复杂 JSON/Dict 优先定义 Pydantic 模型，避免裸 `dict`。

4. **错误处理与降级**
   - 所有外部调用（LLM、数据库、图存储、文件系统）必须 `try/except`；
   - 记录日志后返回可接受的降级结果，不能抛 500。

5. **测试纪律**
   - 新增功能必须配套测试；
   - 提交前运行 `cd backend && .\venv\Scripts\python.exe -m pytest -q`。

6. **安全红线**
   - API Key、密码、JWT Secret 必须走 `.env`；
   - 禁止在代码中硬编码真实密钥。

### 9.2 TypeScript / React 前端规范

1. **组件级注释（每个业务组件顶部必须写）**
   - 说明组件对应的需求；
   - 列出主要 props/hooks/函数；
   - 用 TODO 清单标明已完成/待完成项。

   示例：

   ```typescript
   /**
    * ResourceViewer：个性化资源展示容器。
    *
    * 需求：
    * - 根据当前知识点展示生成的资源包；
    * - 支持文档/导图/练习/审核报告四标签页；
    * - 练习可一键运行到 Pyodide 沙箱。
    *
    * TODO:
    * - [已完成] 四标签页基础渲染
    * - [已完成] 练习自动判题与沙箱运行
    * - [待完成] 认知风格差异化渲染
    */
   ```

2. **命名规范**
   - 组件：`PascalCase`
   - hooks/useStore：`camelCase`，必须以 `use` 开头
   - 类型/接口：`PascalCase`，优先用 `type`/`interface`

3. **类型纪律**
   - 减少 `any`；后端返回类型与 `services/api.ts` 对齐；
   - 通用类型优先放到 `frontend/src/types/`。

4. **状态管理**
   - 局部状态：`useState` / `useReducer`
   - 跨组件状态：`zustand` store

5. **构建纪律**
   - 提交前必须 `cd frontend && npm run build` 通过。

### 9.3 TODO 清单格式

前后端统一使用如下格式，方便全局检索：

```text
TODO:
- [已完成] xxx 功能已实现
- [待完成] yyy 功能待实现（原因/计划）
```

- 已完成的条目**保留**在清单里，不要删除，便于复盘。
- 新增待完成项及时写入，避免口头约定。

### 9.4 Code Review 检查项

审查人必须检查：

- [ ] 新代码是否包含模块/组件级注释与 TODO 清单
- [ ] 是否有未处理的异常或裸 `print`
- [ ] 是否误提交 `.env`、日志、截图
- [ ] 是否影响 Demo 主链路
- [ ] 后端是否补充/更新了测试
- [ ] 前端 `npm run build` 是否通过
