## 二、具体更新内容

### 1. 知识图谱 / 学习路径

- 对齐后端 `GET /api/graph/layout`、`GET /api/graph/path`、个人路径等字段类型，新增 `GraphLayoutResponse`、`PersonalPathResponse`、`GraphLayoutData`、`PersonalPathData` 等类型定义。
- 保留后端图谱与路径数据作为业务来源，同时前端继续负责最终视觉布局。原因是当前后端 layout 坐标在演示画面里节点过少或间距不自然，直接渲染会削弱观感。
- 调整节点详情气泡：统一显示在节点右侧，小箭头指向节点，避免有的左有的右导致画面不稳定。
- 路径节点、当前目标节点保留金色发光/动态波纹效果，用于突出“当前学习路径”。
- 点击节点只展示该节点详情，不再强制拖动画布；进入知识图谱页面时再自动定位到当前路径终点。

### 2. 学习资源页

- 恢复并稳定用户确认过的布局：左侧为当前资源包和六类资源按钮，中间偏右为主要学习资料窗口，右侧为资源检查/版本/反馈区，下方为工作区。
- 修复右侧列与底部工作区重叠的问题，资源反馈区域改为在右侧列内部滚动展示。
- 资源反馈文本框改为常驻显示，不再点击后才出现，避免点击文字反馈时页面底部出现黑色条框。
- 练习题交互修复：
  - 切换练习题时保留每道题独立草稿，不再丢失上一题输入。
  - 初始作答模板增加换行和“在下面开始作答”提示，避免用户直接接在 TODO 注释后面输入。
  - “查看参考答案”改为点一次显示、再点一次隐藏。
  - 提交判题后不再展示原始 JSON，而是格式化为用户可读的判题结果、标准输出、用户输出、错误信息。
  - 后端未返回 `expected_output` 时，不再禁用按钮，而是调用代码运行接口展示程序输出，并提示暂不能自动判题。
- 学习资料文本区继续做了排版优化，避免全黑、小字、密集文本导致阅读疲劳。

### 3. 学习对话

- 对话发送已接入后端会话接口，优先使用流式/对话接口，失败时保留降级路径。
- 修复 AI 回复直接展示对象 JSON 的问题，将 `response_message`、`profile` 等字段拆成人类可读文本。
- 修复手动切换认知风格后，发送消息又被前端自动覆盖为视觉型的问题。
- “查看提示”“继续引导”等按钮已经有前端状态入口，但继续引导的深度和差异化仍主要依赖后端 Socrates 流程返回。
- 麦克风按钮目前依赖浏览器 Web Speech API；若浏览器不支持或权限未开启，只能表现为前端状态切换。

### 4. 代码沙箱

- 代码运行继续接入后端 `/code/execute`。
- 变量快照兼容解析后端可能返回的 `variables`、`locals`、`snapshot` 等字段。
- 当后端变量结构不完整时，前端保留基础兜底解析，尽量避免变量面板一直显示“等待运行”。
- 初始样例代码、样例输出、样例变量仍是前端默认示例，见“仍然静态的部分”。

### 5. 掌握进度

- 热力图、BKT、重新分析、学习统计继续对齐后端接口。
- BKT 展示补充了普通用户能理解的解释，将 `guess`、`slip` 转为“猜对概率”“失误概率”。
- 对后端默认参数增加“默认参数 / 样本不足”的展示语义，避免用户误以为这些百分比一定来自真实学习记录。

### 6. 学习画像

- 学习画像、证据、学习统计等继续使用后端真实字段映射。
- 页面布局做了压缩和重排，减少画像和 Agent 协作下方的大面积空白。
- 目前仍有部分画像建议文案由前端根据字段推断生成，后端若能返回更明确的学习策略会更好。

### 7. Agent 协作

- Agent 列表继续使用后端 trace 映射真实执行链路。
- 修复 Reviewer 长期显示黄色的问题：现在状态颜色按后端状态驱动，运行中为黄色，失败为红色，完成/空闲为绿色。
- 减少响应时间区域前方空白，避免右侧时间看起来像被页面裁掉。

## 三、当前仍然静态 / 前端写死的部分

以下内容不是 bug，而是目前后端还没有完整配置或字段，前端为了保证 Demo 可演示保留的静态壳层或兜底：

1. 左侧导航结构、页面名称、功能入口顺序仍由前端写死。
2. Agent 五个角色名称与基础职责壳层仍由前端固定：Profiler、Navigator、Builder、Reviewer、Socrates。后端目前提供 trace，但没有返回完整 Agent 拓扑配置。
3. 知识图谱最终可视化坐标由前端计算，后端 layout 字段已接入但没有直接作为最终渲染坐标使用。
4. 知识图谱在接口失败或数据不足时仍有前端 fallback 节点、边和默认路径，保证页面不空白。
5. 学习资源页面的空状态文案、按钮名称、部分资源卡片描述由前端固定。
6. 练习题没有 `expected_output` 时，前端只能运行代码并展示输出，无法真正自动判题。
7. 代码沙箱初始示例代码、初始说明、空状态变量说明仍由前端写死。
8. 学习画像中的部分建议文案、学习策略标签由前端根据后端字段推断，不是后端直接返回的个性化策略。
9. 掌握进度中 BKT 的通俗解释、颜色图例解释由前端写死；真实数值来自后端。
10. 学习对话中的快捷提示词、部分继续引导按钮文案由前端固定。
11. 资源版本演进展示目前依赖已有版本接口和前端展示模板；若后端缺少 diff / 触发原因，前端只能展示简化信息。
12. 视觉型、听觉型、动觉型三个按钮目前主要改变前端偏好状态和少量展示倾向，缺少后端返回的强差异化学习策略，因此用户感知不明显。

## 四、需要后端队友补充 / 建议优化的逻辑

### 1. 认知风格按钮需要后端策略支撑

当前“视觉型 / 听觉型 / 动觉型”三个按钮的可见变化不够明显。建议后端在画像或资源生成接口中返回更明确的策略字段，例如：

```json
{
  "cognitive_modality": "visual",
  "modality_strategy": "优先使用结构图、流程图、颜色标注和知识图谱路径",
  "preferred_resource_order": ["mindmap", "document", "case", "exercise"],
  "recommended_next_action": "先查看导图，再完成一道路径节点练习",
  "resource_generation_style": {
    "document_style": "diagram_first",
    "exercise_style": "step_visualization",
    "dialogue_style": "short_with_structure"
  }
}
```

建议差异：

- 视觉型：优先导图、知识图谱、流程图、颜色标注。
- 听觉型：优先讲解稿、对话式解释、可朗读文本、音频脚本。
- 动觉型：优先练习题、代码案例、交互任务、即时反馈。

这样按钮点击后不只是前端高亮，而是能真正影响资源排序、生成内容、对话策略和下一步推荐。

### 2. 练习题需要补全 `expected_output`

目前部分练习题后端没有返回 `expected_output`，前端只能运行代码展示输出，无法自动判题。建议资源生成或练习生成接口保证每道可判题练习至少返回：

- `question`
- `starter_code`
- `expected_output`
- `solution`
- `hints`

### 3. Agent trace 建议返回完整状态语义

目前前端根据 `running`、`failed` 等状态映射颜色。建议后端统一返回：

- `agent`
- `stage`
- `status`
- `duration_ms`
- `message`
- `error`
- `sub_steps`

尤其 Reviewer 内部若包含 Debate / Evaluator / Guardian / Socrates 子流程，建议通过 `sub_steps` 返回，前端才能展示真实协作链，而不是固定壳层。

### 4. 知识图谱 layout 若要后端驱动，需要返回可直接渲染的坐标

当前后端 layout 字段已接入，但实际视觉效果不适合直接展示。若后端希望前端完全按 layout 渲染，需要保证：

- 节点数量完整。
- 节点不会互相遮挡。
- 层级、依赖关系和路径方向清晰。
- 坐标适配大画布，可拖拽浏览。
- 当前路径节点和终点节点可通过字段明确标记。

### 5. BKT 建议返回可信度说明

建议 BKT 接口补充：

- `is_default`
- `sample_count`
- `last_updated`
- `explanation`

这样前端可以明确告诉用户“这是默认参数”还是“这是基于你的练习记录计算出的掌握度”。

### 6. 资源版本演进建议返回结构化 diff

建议后端版本接口增加：

- 每个版本修改了什么。
- 为什么触发重审。
- 错误率或反馈来源。
- 旧版本与新版本差异。

这样知识熔炉 / 版本演进才不会只是时间线展示，而能成为真正的创新亮点。

## 五、剩余风险

- 如果后端当前环境仍是 `LLM_PROVIDER=mock`，学习对话和资源生成会表现为固定回复或模板化内容，这不是前端能完全解决的问题。
- 前端已尽量兼容后端字段，但如果接口字段继续变化，需要同步更新 `frontend/src/services/api.ts` 和相关类型。
- 部分视觉布局目前是为了 Demo 效果前端计算的，如果后端后续提供更优 layout，需要再切换为真正后端驱动。
- 认知风格按钮目前最需要后端策略字段配合，否则用户会继续感觉“点了没什么变化”。

## 六、后续建议

1. 后端优先补齐练习题 `expected_output`，保证练习题能真正判题。
2. 后端补齐认知风格策略字段，让视觉型、听觉型、动觉型真正影响资源生成与推荐。
3. 后端返回更完整的 Agent trace / sub_steps，让 Agent 协作面板从“展示壳层”变成真实流程监控。
4. 前端下一步可集中做 PPT / Demo 视频素材整理，避免继续大规模改动影响稳定性。
